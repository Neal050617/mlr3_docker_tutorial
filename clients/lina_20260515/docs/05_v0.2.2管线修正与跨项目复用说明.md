## 修正和升级摘要

| 问题 | 原因 | 当前处理 | 旧结果是否必须重跑 |
|---|---|---|---|
| `模型比较` sheet 出现 `.x/.y` 或 `_thr` 列 | 阈值派生指标回并到性能表时存在同名列冲突 | Step2 主脚本已在写 Excel 前清理并规范展示 `Sensitivity`、`Specificity`、`Train_Threshold`、`Test_Threshold` | 若只影响 workbook 展示，可用修复脚本；若要获得最新 Step2 行为，建议全量重跑 |
| CV ROC 9 条曲线颜色不可区分 | 绘图时模型标签和颜色映射不稳定，尤其 `rf`/`ranger` 命名差异 | Step2 绘图统一传入模型颜色和标签 | 需要重跑 Step2 或从 RData 重画图 |
| Step1.1-Step1.5 EFS 只有 2 个模型 | EFS learner 集合被压缩，不符合旧版报告口径 | EFS-capable learner 恢复为 `xgb,rf,svm,lgbm` 四个 | 如果要让特征选择过程本身使用 4 个 learner，必须重跑 Step2 |
| SHAP waterfall 只能固定 train/test 样本 | Step3 缺少按组和显式样本选择入口 | 新增 case/control 自动选择、显式样本列表、valid 支持和 force plot | 不必重跑 Step2；可基于已有 `efs.RData` 重跑 Step3 |
| 多个 SHAP 输出目录容易被汇总混在一起 | 汇总脚本过去扫描全部 SHAP 目录 | 新增 `--shap-dir-name` 和项目 wrapper 的 `SHAP_SUMMARY_DIR_NAME` | 重新跑汇总即可 |
| 最终汇总依赖单独 threshold 修正脚本 | wrapper 默认先跑 repair 脚本，容易掩盖 Step2 主脚本是否已修正 | `REFRESH_THRESHOLD_METRICS` 默认 `FALSE`，只有显式设为 `TRUE` 才跑修复脚本 | 新结果应默认不依赖修复脚本；旧结果需要时可显式运行 |

## 其他项目如何处理

### 1. 已完成分析，只想修复展示表头

如果项目只存在 `model_performance_comparison.xlsx` 中 `.x/.y`、`_thr` 列或单模型 sheet 中 `_thr` 行的问题，且不准备改变 EFS 特征选择结果，可以只做 workbook 展示修复：

```bash
Rscript scripts/repair_mlr3_threshold_columns.R \
  --root=/path/to/result_root
```

修复后重新汇总：

```bash
Rscript scripts/summarize_mlr3_results.R \
  --root=/path/to/result_root \
  --tasks=all
```

注意：这不会改变模型训练、特征选择或 SHAP 结果，只是规范 Excel 展示和汇总读取。

### 2. 已完成分析，但要恢复 4 learner EFS 口径

如果项目报告中需要 Step1.1-Step1.5 的 EFS 诊断图和特征选择过程都反映 `xgb,rf,svm,lgbm` 四个模型，应重跑 Step2。只重画诊断图不能改变 `selected_features` 或正式模型训练结果。

重跑后应核对：

```bash
Rscript -e 'root<-"/path/to/result_root"; dirs<-sort(Sys.glob(file.path(root,"run_model_*_step2"))); for (d in dirs) { e<-new.env(parent=emptyenv()); load(file.path(d,"efs.RData"), envir=e); obj<-if (exists("learners_efs", envir=e, inherits=FALSE)) get("learners_efs", envir=e) else list(); ids<-vapply(obj, function(x) if(!is.null(x$id)) x$id else NA_character_, character(1)); ids<-sub("^classif[.]","",ids); cat(basename(d), "\t", paste(ids, collapse=","), "\n", sep="") }'
```

期望每个分支输出：

```text
xgb,rf,svm,lgbm
```

### 3. 已完成 Step2，只想升级 SHAP

可以基于已有 `run_model_*_step2/efs.RData` 单独重跑 Step3。新 SHAP 支持：

- 默认 case/control 各 1 个 waterfall 样本。
- `SHAP_WATERFALL_SAMPLES` 或 `--waterfall_samples` 指定一个或多个样本。
- 样本 ID 支持 `train:ID`、`test:ID`、`valid:ID` 前缀。
- valid 数据存在时可出图；不存在时状态表记录为 `skipped`。
- 同步输出 force plot。

示例：

```bash
SHAP_OUTDIR_NAME=step3_shap_formal_case_control_force_20260526 \
SHAP_DATASETS=train,test,valid \
SHAP_WATERFALL_GROUPS=case,control \
SHAP_WATERFALL_PER_GROUP=1 \
SHAP_LOCAL_DISPLAY_FEATURES=10 \
SHAP_FORCE_PLOTS=TRUE \
bash lina_20260515/scripts/run_gp01_shap_all.sh /path/to/result_root
```

`SHAP_LOCAL_DISPLAY_FEATURES=10` 只限制 waterfall/force 单样本局部解释图；waterfall 保持多行 Top 10 局部贡献图，force 保持较低的箭头形状单行正负累积贡献条。importance、beeswarm、waterfall 和 force 只对较长特征名自动换行，默认超过 28 个字符时优先在靠近中点的 `-` 后断为最多两行；短特征名不换行。SHAP Excel 仍保留完整数值。

其他项目应直接使用仓库级通用脚本：

```bash
bash scripts/run_mlr3_shap_all.sh /path/to/result_root
```

指定样本示例：

```bash
SHAP_WATERFALL_SAMPLES=train:AF33,test:HC7 \
SHAP_FORCE_PLOTS=TRUE \
bash lina_20260515/scripts/run_gp01_shap_all.sh /path/to/result_root
```

其他项目对应写法：


如果同一个结果根目录下保留多个 SHAP 输出目录，汇总时必须指定目录名：

```bash

```


## 解释边界

- `repair_mlr3_threshold_columns.R` 是旧结果的展示修复工具，不应作为新结果正确性的必要条件。
- EFS 恢复为 4 个 learner 会影响特征选择过程；如果项目正式结论依赖入选特征，应优先重跑 Step2，而不是只重画图。
- SHAP 用于解释已训练模型，不用于模型选择。
- 多个 SHAP 输出目录共存时，必须在汇总阶段指定 `--shap-dir-name`，否则容易把 fastcheck/formal/历史输出混在一起。
